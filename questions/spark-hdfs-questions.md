## Что такое hdfs?

**HDFS (Hadoop Distributed File System)** — это распределенная файловая система, созданная для хранения очень больших объемов данных на кластере обычных серверов.

Ключевые особенности:
- **Распределенное хранение**: данные разбиваются на блоки (обычно 128 МБ) и хранятся на разных серверах
- **Отказоустойчивость**: каждый блок реплицируется на несколько серверов (обычно 3 копии)
- **Масштабируемость**: можно добавлять новые узлы для увеличения емкости
- **Высокая пропускная способность**: оптимизирована для потоковой обработки больших данных

Архитектура:
- **NameNode** — главный узел, хранит метаданные (где физически находятся блоки файлов)
- **DataNode** — рабочие узлы, хранят фактические блоки данных

Пример работы:
```bash
# Просмотр файлов в HDFS
hdfs dfs -ls /data/

# Копирование файла из локальной системы в HDFS
hdfs dfs -put localfile.csv /data/

# Чтение содержимого файла
hdfs dfs -cat /data/file.txt
```

## Что такое spark и какие ядра в него входят?

**Apache Spark** — это быстрый универсальный движок для обработки больших данных. Ключевые особенности:
- Выполняет вычисления в памяти (in-memory)
- В 100 раз быстрее Hadoop MapReduce для задач в памяти
- Поддерживает несколько языков программирования (Scala, Java, Python, R)
- Имеет унифицированную платформу для различных типов обработки данных

**Основные компоненты (ядра) Spark**:

1. **Spark Core** — фундаментальный компонент, предоставляет:
   - Базовую функциональность (планирование задач, управление памятью)
   - API для работы с RDD (Resilient Distributed Datasets)
   - Интеграцию с хранилищами данных

2. **Spark SQL** — модуль для работы со структурированными данными:
   - Позволяет выполнять SQL-запросы
   - Поддерживает DataFrame и Dataset API
   - Оптимизирует запросы с помощью Catalyst

   ```python
   # Пример Spark SQL
   df = spark.read.csv("data.csv", header=True)
   result = spark.sql("SELECT count(*) FROM table WHERE age > 25")
   ```

3. **Spark Streaming** — обработка потоковых данных:
   - Разбивает входящий поток на микро-пакеты
   - Обрабатывает с минимальной задержкой
   - Интегрируется с Kafka, Flume, HDFS

   ```python
   # Пример Spark Streaming
   streaming_df = spark.readStream.format("kafka").option("kafka.bootstrap.servers", "host:port").load()
   ```

4. **MLlib** — библиотека машинного обучения:
   - Алгоритмы классификации, регрессии, кластеризации
   - Подготовка данных и оценка моделей
   - Работает на DataFrame API

   ```python
   # Пример MLlib
   from pyspark.ml.classification import LogisticRegression
   lr = LogisticRegression().fit(training_data)
   ```

5. **GraphX** — обработка графов:
   - Создание и преобразование графов
   - Алгоритмы на графах (PageRank, Connected Components)
   - Интеграция между графами и таблицами

   ```scala
   // Пример GraphX (Scala)
   val graph = GraphFrame.fromVertices(vertices, edges)
   val result = graph.pageRank.run()
   ```

6. **Structured Streaming** (новее чем Spark Streaming):
   - Потоковая обработка на основе DataFrame API
   - Обеспечивает end-to-end exactly-once гарантии
   - Поддерживает инкрементальные запросы

## Что такое map reduce?

**MapReduce** — это модель программирования и фреймворк для обработки больших объемов данных на кластере компьютеров.

**Принцип работы**:
1. **Map** — функция, применяемая к каждому элементу входных данных для создания пар ключ-значение
2. **Shuffle** — автоматическая группировка всех значений с одним ключом
3. **Reduce** — функция, объединяющая все значения для каждого ключа

**Простой пример** — подсчет слов в документе:
1. **Map**: для каждого слова выдает пару (слово, 1)
2. **Shuffle**: группирует все единицы с одинаковыми словами
3. **Reduce**: суммирует все единицы для каждого слова

```python
# Пример WordCount в PySpark
text_file = spark.sparkContext.textFile("hdfs:///data/text.txt")
counts = text_file.flatMap(lambda line: line.split(" ")) \
                 .map(lambda word: (word, 1)) \
                 .reduceByKey(lambda a, b: a + b)
counts.saveAsTextFile("hdfs:///output/wordcount")
```

**Преимущества MapReduce**:
- Автоматическое распараллеливание и распределение
- Отказоустойчивость
- Масштабируемость для обработки петабайт данных

**Недостатки**:
- Высокая латентность из-за запись на диск между этапами
- Не подходит для итеративных алгоритмов
- В чистом виде (Hadoop MapReduce) медленнее, чем Spark

## Какие механизмы выполнения операций соединения в спарке вы знаете?

В Spark существует несколько стратегий соединения (join) таблиц:

1. **Broadcast Hash Join** (Broadcast Join):
   - Подходит, когда одна таблица маленькая, а другая большая
   - Маленькая таблица рассылается (broadcast) всем исполнителям
   - Большая таблица остается распределенной
   - Очень эффективен, если одна таблица помещается в память
   - Порог размера таблицы для броадкаста настраивается (spark.sql.autoBroadcastJoinThreshold)

   ```python
   # Явное указание broadcast join
   from pyspark.sql.functions import broadcast
   result = df1.join(broadcast(df2), df1.key == df2.key)
   ```

2. **Shuffle Hash Join**:
   - Используется, когда обе таблицы большие
   - Данные партиционируются (shuffle) по ключу соединения
   - На каждой партиции строится хеш-таблица
   - Эффективен при неравных размерах таблиц, но не таких, чтобы использовать broadcast

3. **Sort Merge Join**:
   - Хорошо подходит для больших таблиц примерно одинакового размера
   - Таблицы сортируются по ключу соединения и объединяются
   - Является стратегией по умолчанию для больших таблиц
   - Не требует много памяти, но требует сортировки

4. **Nested Loop Join**:
   - Самый простой, но наименее эффективный способ
   - Для каждой строки из первой таблицы просматривается вся вторая таблица
   - Используется только когда другие методы не применимы
   - Очень неэффективен для больших данных

Настройка и проверка типа join:
```python
# Установка порога для broadcast join
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 10485760)  # 10MB

# Просмотр плана выполнения для определения типа join
df1.join(df2, df1.key == df2.key).explain()
```

## Что такое спарк сессия и какими параметрами она задается?

**Spark Session** — это единая точка входа для программирования в Spark, объединяющая все предыдущие API контекстов (SparkContext, SQLContext, HiveContext и др.).

**Создание SparkSession**:

```python
# PySpark
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("MyApplication") \
    .master("local[*]") \
    .config("spark.executor.memory", "2g") \
    .config("spark.driver.memory", "1g") \
    .config("spark.sql.shuffle.partitions", "100") \
    .enableHiveSupport() \
    .getOrCreate()
```

**Основные параметры**:

1. **Базовые параметры**:
   - **appName** — имя приложения (отображается в UI)
   - **master** — URL кластера (local, yarn, spark://host:port)
   - **enableHiveSupport** — включение поддержки Hive

2. **Параметры памяти**:
   - **spark.driver.memory** — объем памяти для драйвера
   - **spark.executor.memory** — объем памяти для каждого исполнителя
   - **spark.memory.fraction** — доля памяти для выполнения и хранения (от JVM)
   - **spark.memory.storageFraction** — доля памяти для хранения (от fraction)

3. **Параметры выполнения**:
   - **spark.executor.instances** — количество исполнителей
   - **spark.executor.cores** — количество ядер на исполнителя
   - **spark.default.parallelism** — количество партиций при параллельной обработке
   - **spark.sql.shuffle.partitions** — количество партиций для операций shuffle

4. **Параметры оптимизации**:
   - **spark.sql.autoBroadcastJoinThreshold** — максимальный размер таблицы для broadcast join
   - **spark.sql.adaptive.enabled** — включение адаптивного планирования запросов
   - **spark.serializer** — сериализатор для передачи данных между узлами

5. **Параметры логирования**:
   - **spark.eventLog.enabled** — включение журнала событий
   - **spark.eventLog.dir** — директория для журналов

## В чем отличие спарк сессии от спарк контекста?

**SparkContext** и **SparkSession** — это два разных API для работы со Spark, с несколькими ключевыми отличиями:

**SparkContext**:
- Старый API, появившийся в ранних версиях Spark
- Предоставляет доступ только к RDD API
- Для каждого типа задач требовался отдельный специализированный контекст:
  - SQLContext для SQL-запросов
  - HiveContext для работы с Hive
  - StreamingContext для потоковой обработки
- Создается вручную:
  ```python
  from pyspark import SparkContext, SparkConf
  conf = SparkConf().setAppName("MyApp").setMaster("local")
  sc = SparkContext(conf=conf)
  ```

**SparkSession**:
- Новый унифицированный API, представленный в Spark 2.0
- Объединяет функциональность всех предыдущих контекстов
- Единая точка входа для:
  - RDD (через sparkContext)
  - DataFrame и Dataset
  - SQL-запросов
  - Hive-таблиц
- Более удобное создание:
  ```python
  from pyspark.sql import SparkSession
  spark = SparkSession.builder.appName("MyApp").getOrCreate()
  ```
- Доступ к SparkContext через SparkSession:
  ```python
  sc = spark.sparkContext
  ```

**Основные различия в использовании**:
1. **Доступные API**:
   - SparkContext: только RDD
   - SparkSession: RDD, DataFrame, Dataset, SQL

2. **Удобство использования**:
   - SparkContext: для каждого функционала нужен отдельный контекст
   - SparkSession: все функции доступны через один объект

3. **Версии Spark**:
   - SparkContext: основной в Spark 1.x
   - SparkSession: рекомендуется в Spark 2.x и выше

4. **Типы данных**:
   - SparkContext: работает с необработанными RDD
   - SparkSession: поддерживает структурированные данные и схемы

**Практический совет**: В современных версиях Spark рекомендуется всегда использовать SparkSession как точку входа, а к SparkContext обращаться через нее (spark.sparkContext) при необходимости.

## Как выполняется работа приложения в спарке (workflow)?

**Жизненный цикл Spark-приложения** включает следующие этапы:

1. **Инициализация приложения**:
   - Создается SparkSession/SparkContext
   - Приложение регистрируется в кластерном менеджере (YARN, Mesos, K8s)
   - Выделяются ресурсы для драйвера

2. **Создание логического плана**:
   - В коде определяются операции над данными (transformations)
   - Spark строит направленный ациклический граф (DAG) операций
   - Операции не выполняются, пока не вызвано действие (action)

3. **Оптимизация и планирование**:
   - При вызове действия (action) Spark оптимизирует план выполнения
   - Логические операции преобразуются в физический план
   - Применяются оптимизации (отсечение партиций, объединение операций)
   - План разбивается на стадии (stages), разделенные операциями shuffle

4. **Выделение ресурсов**:
   - Кластерный менеджер выделяет исполнителей (executors)
   - Каждый исполнитель получает выделенные CPU и память
   - Память разделяется на execution, storage и user memory

5. **Выполнение задач**:
   - Каждая стадия делится на задачи (tasks)
   - Задачи распределяются по исполнителям
   - Драйвер координирует выполнение

6. **Управление памятью**:
   - RDD кэшируются в памяти, если используются повторно
   - При нехватке памяти применяются стратегии вытеснения (LRU)
   - Неиспользуемые данные удаляются сборщиком мусора

7. **Возврат результатов**:
   - Результаты собираются на драйвере или записываются во внешнее хранилище
   - Освобождаются ресурсы (по завершении или при ошибке)

**Пример с разбивкой workflow**:
```python
# 1. Инициализация приложения
spark = SparkSession.builder.appName("Example").getOrCreate()

# 2. Создание логического плана (не выполняется на этом этапе)
df = spark.read.csv("data.csv")
filtered = df.filter(df.age > 25)
grouped = filtered.groupBy("department").count()

# 3-6. Триггер выполнения (здесь начинается реальная работа)
result = grouped.collect()  # action - запускает выполнение

# 7. Завершение
spark.stop()
```

**Ключевые моменты по ресурсам**:
- Память выделяется при запуске приложения, а не динамически
- Задачи распределяются по доступным исполнителям
- При нехватке ресурсов происходит откат к дисковому хранению

## Что такое преобразование и действие в спарке?

В Spark все операции делятся на два типа: преобразования (Transformations) и действия (Actions). Это ключевая концепция для понимания ленивых вычислений в Spark.

**Преобразования (Transformations)**:
- Создают новый RDD/DataFrame из существующего
- **Ленивые** — не выполняются сразу, только запоминаются
- Формируют направленный ациклический граф (DAG) операций
- Примеры преобразований:
  - `map()`, `filter()`, `flatMap()`
  - `groupByKey()`, `reduceByKey()`
  - `join()`, `union()`
  - `select()`, `where()`, `groupBy()`

```python
# Примеры преобразований (не выполняются на этом этапе)
filtered_rdd = rdd.filter(lambda x: x > 10)
mapped_df = df.select("name", "age").filter(df.age > 21)
```

**Действия (Actions)**:
- Запускают выполнение всех накопленных преобразований
- Возвращают результат в драйвер или записывают его
- Примеры действий:
  - `count()`, `collect()`, `first()`
  - `take(n)`, `foreach()`
  - `saveAsTextFile()`, `show()`
  - `write.csv()`, `write.parquet()`

```python
# Примеры действий (запускают выполнение)
result = filtered_rdd.collect()  # возвращает все элементы как список
count = mapped_df.count()        # возвращает количество строк
mapped_df.write.parquet("data")  # сохраняет результат в Parquet
```

**Узкие vs широкие преобразования**:
- **Узкие преобразования** (narrow transformations):
  - Каждая входная партиция влияет только на одну выходную
  - Не требуют перемешивания данных (shuffle)
  - Выполняются быстрее, могут применяться конвейерно
  - Примеры: map, filter, flatMap

- **Широкие преобразования** (wide transformations):
  - Данные из разных партиций комбинируются
  - Требуют перемешивания данных (shuffle) по сети
  - Создают границы между стадиями в плане выполнения
  - Примеры: groupByKey, reduceByKey, join, distinct

**Практические моменты**:
- Ленивые вычисления позволяют Spark оптимизировать план выполнения
- Старайтесь минимизировать количество действий и широких преобразований
- Цепочка узких преобразований выполняется за один проход данных
- При отладке полезно использовать `.persist()` или `.cache()` для сохранения промежуточных результатов

## Что такое драйвер и экзекьютор?

**Драйвер (Driver)** и **Исполнитель (Executor)** — это два основных компонента архитектуры выполнения Spark-приложения.

**Драйвер (Driver)**:
- Процесс, в котором запускается основной код приложения
- Отвечает за:
  - Преобразование кода в задачи (tasks)
  - Планирование и координацию выполнения задач
  - Сбор результатов от исполнителей
  - Взаимодействие с кластерным менеджером
- Содержит SparkContext/SparkSession
- Требования к ресурсам настраиваются через параметры `spark.driver.memory` и `spark.driver.cores`
- Обычно запускается на машине, откуда запущено приложение (client mode) или на узле кластера (cluster mode)

**Исполнитель (Executor)**:
- Рабочие процессы, запускаемые на узлах кластера
- Отвечают за:
  - Выполнение задач, назначенных драйвером
  - Хранение кэшированных данных RDD/DataFrame в памяти
  - Возврат результатов драйверу
- Запускаются при старте приложения и живут до его завершения
- Могут выполнять несколько задач параллельно
- Настраиваются через параметры:
  - `spark.executor.instances` — количество исполнителей
  - `spark.executor.memory` — память для каждого исполнителя
  - `spark.executor.cores` — количество ядер CPU на исполнителя

**Взаимодействие Драйвера и Исполнителей**:

1. **Запуск приложения**:
   - Драйвер запрашивает ресурсы у кластерного менеджера (YARN, Kubernetes)
   - Кластерный менеджер запускает исполнителей на рабочих узлах

2. **Выполнение**:
   - Драйвер разбивает DAG операций на стадии и задачи
   - Драйвер отправляет задачи исполнителям
   - Исполнители выполняют задачи и сообщают о результатах драйверу
   - Драйвер контролирует зависимости между стадиями

3. **Отказоустойчивость**:
   - Драйвер отслеживает состояние исполнителей
   - При сбое исполнителя задачи перепланируются на другие исполнители
   - При сбое драйвера все приложение останавливается

**Схема архитектуры**:
```
[Кластерный менеджер (YARN/Mesos/Kubernetes)]
         ↑↓
[Драйвер (SparkContext/SparkSession)]
   ↓↑       ↓↑       ↓↑
[Executor 1] [Executor 2] ... [Executor N]
```

**Рекомендации**:
- Для больших данных увеличивайте количество исполнителей, а не их размер
- Оптимально выделять 4-8 ядер на исполнителя
- Установите память драйвера достаточной для сбора всех результатов
- В YARN лучше работает режим cluster, где драйвер запускается внутри кластера

## Что такое rdd, dataset и dataframe?

**RDD (Resilient Distributed Dataset)**:
- Низкоуровневая, базовая абстракция данных в Spark
- **Особенности**:
  - Неизменяемая распределенная коллекция объектов
  - Разделена на партиции, обрабатываемые параллельно
  - Отказоустойчивая (хранит информацию о происхождении — lineage)
  - Типобезопасная на уровне языка (компиляции)
  - Операции на уровне объектов (нет оптимизации на уровне SQL)
- **Когда использовать**:
  - Для неструктурированных данных
  - Когда требуется низкоуровневый контроль
  - Для работы со сложными типами данных
- **Пример**:
  ```python
  # Создание RDD
  rdd = sc.parallelize([1, 2, 3, 4, 5])
  
  # Операции с RDD
  result = rdd.filter(lambda x: x > 2) \
              .map(lambda x: x * 2) \
              .reduce(lambda a, b: a + b)
  ```

**DataFrame**:
- Структурированная коллекция данных, организованная в именованные столбцы
- **Особенности**:
  - Похож на таблицу в реляционной базе данных
  - Имеет схему (структуру столбцов и их типов)
  - Оптимизируется Catalyst оптимизатором
  - Не является типобезопасным на уровне компиляции
  - Аналог R/Pandas DataFrame, но распределенный
- **Когда использовать**:
  - Для структурированных данных
  - При работе с SQL-запросами
  - Для интеграции с инструментами BI
- **Пример**:
  ```python
  # Создание DataFrame
  df = spark.read.csv("data.csv", header=True, inferSchema=True)
  
  # Операции с DataFrame
  result = df.filter(df.age > 25) \
             .groupBy("department") \
             .agg({"salary": "avg", "age": "max"}) \
             .show()
  ```

**Dataset**:
- Гибрид DataFrame и RDD, доступен в Java и Scala (в Python нет, только DataFrame)
- **Особенности**:
  - Типобезопасность на уровне компиляции
  - Оптимизации высокоуровневого API (Catalyst, Tungsten)
  - Строго типизированные объекты и операции
  - Легкое преобразование между RDD и Dataset
- **Когда использовать**:
  - В Scala/Java для сложных агрегаций и трансформаций
  - Когда нужна типобезопасность и оптимизация запросов
  - Для сочетания преимуществ RDD и DataFrame
- **Пример** (Scala):
  ```scala
  // Определение класса
  case class Person(name: String, age: Int)
  
  // Создание Dataset
  val dataset = spark.read
    .option("header", "true")
    .csv("people.csv")
    .as[Person]
  
  // Операции с Dataset
  val result = dataset
    .filter(p => p.age > 21)
    .map(p => p.name)
    .collect()
  ```

**Как RDD работает «под капотом»**:
1. **Партицирование**:
   - Данные разделяются на части (партиции)
   - Каждая партиция обрабатывается параллельно

2. **Отказоустойчивость**:
   - RDD отслеживает происхождение данных (lineage)
   - При сбое узла партиции восстанавливаются по записанному графу операций

3. **Ленивые вычисления**:
   - Операции не выполняются, пока не будет вызвано действие (action)
   - Позволяет оптимизировать план выполнения

4. **Кэширование**:
   - Промежуточные RDD могут быть сохранены в памяти или на диске
   - Ускоряет повторное использование данных

5. **Сериализация**:
   - Объекты сериализуются для передачи между узлами
   - Выбор сериализатора влияет на производительность

**Эволюция абстракций данных в Spark**:
- RDD (Spark 1.0) → DataFrame (Spark 1.3) → Dataset (Spark 1.6)
- В современных приложениях рекомендуется использовать DataFrame/Dataset API
- RDD остается полезным для низкоуровневого доступа и сложных преобразований

## В чем разница форматов avro, parquet, orc?

**Avro, Parquet и ORC** — это форматы хранения данных для больших распределенных систем, каждый со своими особенностями.

**Avro**:
- **Формат**: строково-ориентированный (row-based)
- **Особенности**:
  - Схема хранится вместе с данными
  - Поддерживает эволюцию схемы (добавление/удаление полей)
  - Эффективная сериализация и хорошее сжатие
  - Поддерживает динамическую типизацию
- **Преимущества**:
  - Отлично подходит для потоковой обработки и запись/дополнение
  - Хорошая производительность записи
  - Компактное бинарное представление данных
- **Недостатки**:
  - Менее эффективен для аналитических запросов
  - Необходимость чтения всех записей
- **Когда использовать**:
  - Для потоковой передачи данных
  - Когда схема часто меняется
  - Для систем обмена сообщениями (Kafka)

**Parquet**:
- **Формат**: колоночно-ориентированный (column-based)
- **Особенности**:
  - Эффективное хранение и сжатие данных
  - Поддерживает вложенные структуры данных
  - Оптимизирован для запросов, читающих подмножество столбцов
  - Хранит статистику и метаданные для каждого блока
- **Преимущества**:
  - Высокая эффективность запросов и фильтрации
  - Отличное сжатие данных
  - Поддержка предикативной фильтрации (pushdown)
- **Недостатки**:
  - Менее эффективен для полной записи/обновления данных
  - Сложнее для потоковой обработки
- **Когда использовать**:
  - Для аналитических запросов
  - Когда запросы выбирают подмножество столбцов
  - Для хранилищ данных и озер данных

**ORC (Optimized Row Columnar)**:
- **Формат**: колоночно-ориентированный, оптимизированный для Hive и Hadoop
- **Особенности**:
  - Хранит индексы внутри файлов для быстрого пропуска блоков
  - Интегрированное индексирование и статистика
  - Поддержка ACID транзакций в Hive
  - Эффективное сжатие на уровне столбцов
- **Преимущества**:
  - Очень эффективные запросы аналитики
  - Меньший размер файлов чем Parquet в некоторых случаях
  - Хорошо работает с Hive
- **Недостатки**:
  - Исторически тесно связан с экосистемой Hadoop
  - Меньшая поддержка в некоторых инструментах
- **Когда использовать**:
  - В экосистеме Hive/Hadoop
  - Для аналитических запросов с большой селективностью
  - Для кейсов с транзакционными требованиями в Hive

**Сравнение производительности**:
- **Запись данных**: Avro > Parquet > ORC
- **Чтение всех данных**: Avro ≈ Parquet ≈ ORC
- **Чтение подмножества столбцов**: ORC > Parquet > Avro
- **Сжатие**: ORC ≈ Parquet > Avro

**Пример использования в Spark**:
```python
# Чтение/запись в формате Avro
df.write.format("avro").save("data.avro")
avro_df = spark.read.format("avro").load("data.avro")

# Чтение/запись в формате Parquet
df.write.parquet("data.parquet")
parquet_df = spark.read.parquet("data.parquet")

# Чтение/запись в формате ORC
df.write.orc("data.orc")
orc_df = spark.read.orc("data.orc")
```

## Что такое hdfs dfs?

**hdfs dfs** (или просто **dfs**) — это командная утилита для работы с распределенной файловой системой Hadoop (HDFS) из командной строки.

**Назначение**:
- Управление файлами и директориями в HDFS
- Копирование файлов между локальной файловой системой и HDFS
- Просмотр содержимого файлов в HDFS
- Получение информации о файлах и пространстве

**Базовый синтаксис**:
```bash
hdfs dfs -command [arguments]
```

или (эквивалентно):

```bash
hadoop fs -command [arguments]
```

**Основные команды**:

1. **Навигация и просмотр**:
   - **-ls [path]** — список файлов/папок
   - **-cat [file]** — просмотр содержимого файла
   - **-tail [file]** — последние строки файла
   - **-du [path]** — размер файла/директории
   - **-count [path]** — количество файлов/директорий

   ```bash
   hdfs dfs -ls /user/data
   hdfs dfs -cat /user/data/file.txt
   ```

2. **Управление файлами**:
   - **-mkdir [path]** — создание директории
   - **-rm [path]** — удаление файла
   - **-rmr [path]** — рекурсивное удаление
   - **-mv [src] [dst]** — перемещение файла
   - **-cp [src] [dst]** — копирование файла
   - **-chmod [mode] [path]** — изменение прав доступа
   - **-chown [owner] [path]** — изменение владельца

   ```bash
   hdfs dfs -mkdir /user/data/new_folder
   hdfs dfs -rm /user/data/old_file.txt
   ```

3. **Копирование между локальной ФС и HDFS**:
   - **-put [local-src] [hdfs-dst]** — копирование из локальной в HDFS
   - **-get [hdfs-src] [local-dst]** — копирование из HDFS в локальную
   - **-copyFromLocal [local-src] [hdfs-dst]** — аналог put
   - **-copyToLocal [hdfs-src] [local-dst]** — аналог get

   ```bash
   hdfs dfs -put local_file.csv /user/data/
   hdfs dfs -get /user/data/report.csv ./local_report.csv
   ```

4. **Дополнительные команды**:
   - **-df -h** — свободное пространство в HDFS
   - **-setrep -w 3 [path]** — установка фактора репликации
   - **-test -e [path]** — проверка существования файла
   - **-checksum [file]** — вычисление контрольной суммы

   ```bash
   hdfs dfs -df -h
   hdfs dfs -setrep -w 3 /user/data/important_file.txt
   ```

**Практические примеры**:

```bash
# Копирование всех файлов из локальной директории в HDFS
hdfs dfs -put ./local_data/* /user/hadoop/data/

# Вывод содержимого директории с подробной информацией
hdfs dfs -ls -R /user/hadoop/

# Поиск файлов по шаблону
hdfs dfs -find /user -name "*.log"

# Объединение файлов и сохранение результата
hdfs dfs -getmerge /user/logs/*.log combined_logs.txt

# Проверка занимаемого пространства с человекочитаемым форматом
hdfs dfs -du -s -h /user/hadoop/
```

**Дополнительно**:
- Команда `hdfs dfs` почти идентична устаревшей команде `hadoop fs`
- В Hadoop 3.x рекомендуется использовать `hdfs dfs`
- Большинство команд имеют синтаксис, похожий на Unix-команды
- Поддерживается использование групповых символов (wildcards): *, ?

## Как просмотреть статус кластера?

Существует несколько способов мониторинга и проверки статуса Hadoop/Spark кластера:

**1. Веб-интерфейсы**:
   
- **HDFS NameNode UI** (обычно порт 9870 или 50070 в старых версиях):
  - Показывает общую информацию о файловой системе
  - Список DataNode и их статус
  - Использование пространства и распределение блоков
  - URL: `http://<namenode-host>:9870`

- **YARN ResourceManager UI** (обычно порт 8088):
  - Информация о ресурсах кластера (CPU, память)
  - Список запущенных и завершенных приложений
  - Статус узлов (NodeManager)
  - URL: `http://<resourcemanager-host>:8088`

- **Spark History Server** (обычно порт 18080):
  - История выполнения Spark-приложений
  - Детальная информация о стадиях и задачах
  - Метрики производительности
  - URL: `http://<history-server-host>:18080`

- **Spark Master UI** (для Standalone кластера, порт 8080):
  - Статус работников (workers)
  - Информация о запущенных приложениях
  - URL: `http://<spark-master-host>:8080`

**2. Командная строка**:

- **Статус HDFS**:
  ```bash
  # Общая информация о файловой системе
  hdfs dfsadmin -report
  
  # Проверка здоровья HDFS
  hdfs fsck /
  
  # Информация о балансировке блоков
  hdfs balancer -report
  ```

- **Статус YARN**:
  ```bash
  # Информация о кластере
  yarn node -list
  
  # Статус приложений
  yarn application -list
  
  # Детальная информация об определенном приложении
  yarn application -status <application_id>
  ```

- **Статус Spark (для standalone кластера)**:
  ```bash
  # Проверка статуса Master
  $SPARK_HOME/sbin/spark-daemon.sh status org.apache.spark.deploy.master.Master 1
  
  # Проверка статуса Worker
  $SPARK_HOME/sbin/spark-daemon.sh status org.apache.spark.deploy.worker.Worker 1
  ```

**3. Мониторинг через API**:

- **HDFS API**:
  ```bash
  # Получение информации через REST API (требует jq для форматирования JSON)
  curl -s http://<namenode-host>:9870/jmx | jq
  ```

- **YARN API**:
  ```bash
  # Получение информации о кластере
  curl -s http://<resourcemanager-host>:8088/ws/v1/cluster/info | jq
  
  # Получение метрик
  curl -s http://<resourcemanager-host>:8088/ws/v1/cluster/metrics | jq
  ```

**4. Инструменты мониторинга**:

- **Prometheus + Grafana**:
  - Сбор метрик с кластера через JMX экспортер
  - Настраиваемые дашборды для визуализации метрик
  - Настройка алертов

- **Ambari/Cloudera Manager**:
  - Комплексное управление кластером Hadoop
  - Мониторинг всех компонентов
  - Уведомления о проблемах
  - Визуализация метрик

- **Ganglia**:
  - Мониторинг системных ресурсов
  - Исторические данные и тренды
  - Легкая интеграция с Hadoop

**5. Проверка логов**:

- **HDFS логи**:
  ```bash
  # Логи NameNode
  cat /var/log/hadoop/hdfs/hadoop-hdfs-namenode-*.log
  
  # Логи DataNode
  cat /var/log/hadoop/hdfs/hadoop-hdfs-datanode-*.log
  ```

- **YARN логи**:
  ```bash
  # Логи ResourceManager
  cat /var/log/hadoop/yarn/yarn-yarn-resourcemanager-*.log
  
  # Логи NodeManager
  cat /var/log/hadoop/yarn/yarn-yarn-nodemanager-*.log
  ```

- **Spark логи**:
  ```bash
  # Логи приложения (через YARN)
  yarn logs -applicationId <application_id>
  
  # Логи драйвера/исполнителей (standalone)
  cat /var/log/spark/spark-*-*.log
  ```

## Джоба работала 7 дней, потом упала, заказчик не получил результат. В чем проблема?

Если Spark-джоба работала продолжительное время (7 дней), а затем упала без получения результата, необходимо провести комплексное расследование. Вот структурированный подход к диагностике подобных проблем:

**1. Начните с логов**:
- **Лог приложения Spark**:
  ```bash
  # Просмотр логов приложения через YARN
  yarn logs -applicationId <application_id>
  
  # Или через Spark History Server
  http://<history-server>:18080/history/<application_id>
  ```
- **Ищите в логах**:
  - Сообщения об ошибках и исключениях (Exception, Error)
  - Последние события перед сбоем
  - Out of Memory ошибки
  - Java stack traces

**2. Проанализируйте ресурсы**:
- **Память**:
  - Проверьте, не закончилась ли память на исполнителях (OOM errors)
  - Посмотрите, не было ли GC (Garbage Collection) паузы
  - Проверьте настройки `spark.executor.memory` и `spark.driver.memory`

- **Диск**:
  - Проверьте, не закончилось ли место на диске (особенно для shuffle/spill)
  - Посмотрите настройки `spark.local.dir` и доступное место

- **Сеть**:
  - Проверьте логи на предмет ошибок сети или тайм-аутов
  - Посмотрите, не было ли проблем с соединением между узлами

**3. Изучите план выполнения**:
- **UI Spark**:
  - Изучите план выполнения (DAG visualization)
  - Найдите стадии, на которых произошла ошибка
  - Проанализируйте самые долгие и ресурсоемкие стадии

- **SQL Plan**:
  - Если использовались Spark SQL запросы, проверьте план запроса
  - Ищите операции, которые могут вызывать проблемы (shuffle, join огромных таблиц)

**4. Проверьте данные**:
- **Skew (перекос) данных**:
  - Неравномерно распределенные данные могут привести к тому, что некоторые партиции будут намного больше других
  - Проверьте статистику партиций на UI

- **Объем данных**:
  - Возможно, набор данных слишком велик для текущей конфигурации
  - Проверьте, растет ли размер данных с течением времени

**5. Проверьте кластер**:
- **Здоровье узлов**:
  - Проверьте, не вышел ли из строя какой-либо узел во время выполнения
  - Логи YARN/HDFS могут содержать информацию о проблемных узлах

- **Конкуренция за ресурсы**:
  - Проверьте, не было ли других ресурсоемких задач в то же время
  - Возможно, кластер был перегружен

**6. Типичные причины долгих джоб, которые затем падают**:

- **Утечки памяти**:
  - Постепенное накопление объектов в памяти
  - Неэффективное использование колекций
  - Непродуманный broadcast больших объектов

- **Shuffle проблемы**:
  - Слишком большой shuffle, требующий много места на диске
  - Timeout соединений при длительном shuffle

- **Накопление проблем**:
  - Постепенное исчерпание ресурсов (диск, сеть)
  - Медленная деградация производительности

- **Внешние проблемы**:
  - Обслуживание кластера или сбои сети
  - Ошибки в внешних сервисах (HDFS, Hive)

**7. Возможные решения**:

- **Увеличение ресурсов**:
  - Больше исполнителей или памяти для них
  - Больше CPU для длительных операций

- **Оптимизация кода**:
  - Разбиение большой задачи на меньшие под-задачи
  - Использование оконных функций вместо группировок
  - Правильный выбор join-стратегии

- **Реструктуризация данных**:
  - Партиционирование данных для предотвращения skew
  - Предварительная агрегация данных

- **Настройка Spark**:
  - Оптимизация параметров shuffle
  - Настройка сериализации
  - Управление параллелизмом

**8. Рекомендации для мониторинга длительных джоб**:
