## Что такое hive?

**Apache Hive** — это система хранилища данных, построенная поверх Hadoop, которая позволяет анализировать большие наборы данных с использованием SQL-подобного языка (HiveQL).

**Ключевые особенности**:
- Предоставляет SQL-интерфейс для работы с данными в HDFS
- Преобразует SQL-запросы в MapReduce, Tez или Spark задачи
- Хранит метаданные (схемы таблиц) в реляционной базе данных (Metastore)
- Поддерживает разные форматы данных (текст, Parquet, ORC, Avro)
- Позволяет создавать пользовательские функции (UDF)

**Архитектура Hive**:
- **Драйвер** — обрабатывает запросы и планирует их выполнение
- **Metastore** — хранит метаданные таблиц и баз данных
- **Execution Engine** — выполняет план запроса (через MR, Tez, Spark)

**Пример использования**:
```sql
-- Создание таблицы
CREATE TABLE employees (
  id INT,
  name STRING,
  department STRING,
  salary DOUBLE
)
PARTITIONED BY (year INT, month INT)
STORED AS PARQUET;

-- Загрузка данных
LOAD DATA INPATH '/user/data/employee_data.csv' 
INTO TABLE employees
PARTITION (year=2023, month=3);

-- Запрос данных
SELECT department, AVG(salary) as avg_salary
FROM employees
WHERE year = 2023
GROUP BY department
HAVING avg_salary > 5000;
```

**Преимущества**:
- Знакомый SQL-интерфейс для работы с большими данными
- Масштабируемость благодаря Hadoop
- Поддержка различных форматов данных и сжатия
- Интеграция с другими инструментами экосистемы Hadoop

**Недостатки**:
- Высокая латентность (не для OLTP)
- Ограниченная поддержка транзакций
- Не заменяет традиционные СУБД для оперативной обработки

## В чем разница бакетирования и партицирования в hive?

**Партиционирование** и **бакетирование** — два метода организации данных в Hive, которые помогают оптимизировать запросы, но работают по-разному.

**Партиционирование (Partitioning)**:
- **Принцип**: разделение таблицы на части по значению определенного столбца
- **Физическая организация**: каждая партиция хранится в отдельной поддиректории таблицы
- **Назначение**: 
  - Позволяет пропускать чтение ненужных данных (partition pruning)
  - Улучшает производительность фильтрации по партиционным полям
- **Когда использовать**:
  - Для столбцов с низкой кардинальностью (дата, регион, статус)
  - Когда данные естественно распределяются по значению (год, месяц)
  - Для часто используемых условий WHERE

**Синтаксис партиционирования**:
```sql
-- Создание партиционированной таблицы
CREATE TABLE sales (
  transaction_id INT,
  product_id INT,
  amount DOUBLE
)
PARTITIONED BY (year INT, month INT);

-- Вставка данных в конкретную партицию
INSERT INTO sales PARTITION (year=2023, month=3)
SELECT transaction_id, product_id, amount 
FROM sales_source
WHERE year=2023 AND month=3;
```

**Бакетирование (Bucketing)**:
- **Принцип**: распределение данных на фиксированное количество корзин (buckets) с помощью хеш-функции
- **Физическая организация**: данные в каждом бакете хранятся в отдельном файле
- **Назначение**: 
  - Оптимизация join-операций (map-side joins)
  - Более равномерное распределение данных
  - Эффективная выборка образцов данных (sampling)
- **Когда использовать**:
  - Для колонок с высокой кардинальностью (ID, email)
  - Для часто используемых join-условий
  - Когда требуется эффективная сортировка данных

**Синтаксис бакетирования**:
```sql
-- Создание бакетированной таблицы
CREATE TABLE users (
  user_id INT,
  username STRING,
  email STRING
)
CLUSTERED BY (user_id) INTO 16 BUCKETS;

-- Включение бакетирования при загрузке данных
SET hive.enforce.bucketing = true;

-- Загрузка данных (при вставке автоматически распределяется по бакетам)
INSERT INTO users SELECT * FROM users_source;
```

**Ключевые различия**:

1. **Механизм организации**:
   - Партиционирование: физическое разделение по директориям
   - Бакетирование: хеширование данных в фиксированное число файлов

2. **Гранулярность**:
   - Партиционирование: обычно грубое разделение (по дате, региону)
   - Бакетирование: более мелкое распределение (по ID пользователя)

3. **Количество разделов**:
   - Партиционирование: может быть много партиций (тысячи)
   - Бакетирование: обычно фиксированное небольшое количество (8-128)

4. **Основные преимущества**:
   - Партиционирование: пропуск ненужных данных при запросах
   - Бакетирование: оптимизация соединений и работа с выборками

**Совместное использование**:
```sql
CREATE TABLE web_logs (
  request_id STRING,
  ip STRING,
  url STRING,
  timestamp STRING
)
PARTITIONED BY (date STRING)
CLUSTERED BY (ip) SORTED BY (timestamp) INTO 32 BUCKETS;
```

Часто партиционирование и бакетирование используются вместе, создавая многоуровневую систему организации данных для оптимальной производительности.

## Что такое yarn?

**YARN (Yet Another Resource Negotiator)** — это система управления ресурсами и планирования заданий в кластере Hadoop.

**Основное назначение YARN**:
- Управление вычислительными ресурсами кластера (CPU, память)
- Планирование и выполнение различных типов приложений (не только MapReduce)
- Обеспечение многопользовательской работы и изоляции задач
- Повышение эффективности использования ресурсов кластера

**Архитектура YARN**:

1. **ResourceManager (RM)** — главный компонент:
   - Единственный на кластер (с резервной копией для высокой доступности)
   - Принимает запросы на выделение ресурсов
   - Планирует запуск приложений на узлах
   - Контролирует использование ресурсов всего кластера
   - Содержит планировщик (Scheduler)

2. **NodeManager (NM)** — агент на каждом узле:
   - Запускается на каждом рабочем узле кластера
   - Отвечает за контейнеры, выполняющиеся на узле
   - Мониторит использование ресурсов (CPU, память, диск, сеть)
   - Отправляет отчеты ResourceManager о состоянии узла

3. **ApplicationMaster (AM)** — управляющий процесс приложения:
   - Создается для каждого приложения
   - Запрашивает ресурсы у ResourceManager
   - Координирует выполнение приложения
   - Мониторит прогресс и состояние приложения

4. **Container** — исполнительный модуль:
   - Выделенный набор ресурсов (CPU, память) на узле
   - Среда для выполнения задач приложения
   - Изолированный от других контейнеров

**Жизненный цикл приложения в YARN**:

1. Клиент отправляет запрос на запуск приложения ResourceManager
2. ResourceManager выделяет контейнер для ApplicationMaster
3. ApplicationMaster запускается в этом контейнере
4. ApplicationMaster запрашивает ресурсы у ResourceManager
5. ResourceManager выделяет контейнеры согласно доступным ресурсам
6. ApplicationMaster запускает задачи в выделенных контейнерах
7. По завершении, ApplicationMaster сообщает результат и освобождает ресурсы

**Планировщики YARN**:
- **Capacity Scheduler** (по умолчанию): 
  - Поддерживает многопользовательскую работу через очереди
  - Гарантирует минимальные ресурсы для каждой очереди
  - Позволяет разделить кластер между отделами/группами
- **Fair Scheduler**: 
  - Равномерно распределяет ресурсы между всеми пользователями
  - Динамически перераспределяет ресурсы по мере необходимости
- **Custom Schedulers**:
  - Возможность создания собственных планировщиков

**Пример конфигурации YARN для Spark**:
```bash
# Настройка Spark для работы с YARN
spark-submit \
  --master yarn \
  --deploy-mode cluster \
  --executor-memory 2G \
  --num-executors 10 \
  --executor-cores 2 \
  --queue production \
  --name "My Spark App" \
  my_application.py
```

**Преимущества YARN**:
- Мультитенантность (несколько пользователей/приложений)
- Эффективное использование ресурсов
- Поддержка разных типов рабочих нагрузок (Spark, Flink, MapReduce)
- Масштабируемость до тысяч узлов

## Какие параметры прописываются в спарк сессию при использовании yarn?

При использовании YARN в качестве кластерного менеджера для Spark необходимо настроить ряд параметров в SparkSession. Ниже приведены основные параметры, которые задаются при работе с YARN:

**Базовые параметры подключения к YARN**:

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \                      # Указывает YARN как кластерный менеджер
    .config("spark.submit.deployMode", "cluster") \  # "cluster" или "client"
    .getOrCreate()
```

**Параметры распределения ресурсов**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Количество исполнителей (executor-ов)
    .config("spark.executor.instances", "10") \
    # Память для каждого исполнителя
    .config("spark.executor.memory", "4g") \
    # Накладные расходы памяти (overhead)
    .config("spark.executor.memoryOverhead", "1g") \
    # Количество ядер для каждого исполнителя
    .config("spark.executor.cores", "2") \
    # Память для драйвера
    .config("spark.driver.memory", "2g") \
    # Накладные расходы памяти для драйвера
    .config("spark.driver.memoryOverhead", "512m") \
    # Количество ядер для драйвера
    .config("spark.driver.cores", "1") \
    .getOrCreate()
```

**Параметры планирования и очередей YARN**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Указание очереди YARN
    .config("spark.yarn.queue", "production") \
    # Возможность запрашивать дополнительные исполнители динамически
    .config("spark.dynamicAllocation.enabled", "true") \
    # Минимальное количество исполнителей при динамическом распределении
    .config("spark.dynamicAllocation.minExecutors", "5") \
    # Максимальное количество исполнителей при динамическом распределении
    .config("spark.dynamicAllocation.maxExecutors", "20") \
    # Период простоя после которого исполнитель будет убран
    .config("spark.dynamicAllocation.executorIdleTimeout", "60s") \
    .getOrCreate()
```

**Параметры для расшаривания файлов и библиотек**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Архив с Python библиотеками (для PySpark)
    .config("spark.yarn.dist.archives", "hdfs:///user/libs/pyspark_env.zip#environment") \
    # Дополнительные JAR-файлы
    .config("spark.yarn.dist.jars", "hdfs:///user/libs/custom.jar") \
    # Дополнительные файлы (конфиги, словари и т.д.)
    .config("spark.yarn.dist.files", "hdfs:///user/configs/app.conf") \
    # Python для исполнителей
    .config("spark.yarn.appMasterEnv.PYSPARK_PYTHON", "./environment/bin/python") \
    .getOrCreate()
```

**Параметры логирования и мониторинга**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Включение записи событий для History Server
    .config("spark.eventLog.enabled", "true") \
    # Директория для логов событий
    .config("spark.eventLog.dir", "hdfs:///spark-history") \
    # Уровень логирования для драйвера
    .config("spark.driver.extraJavaOptions", "-Dlog4j.rootCategory=INFO") \
    # Уровень логирования для исполнителей
    .config("spark.executor.extraJavaOptions", "-Dlog4j.rootCategory=INFO") \
    .getOrCreate()
```

**Параметры отказоустойчивости**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Максимальное количество попыток запуска задачи
    .config("spark.task.maxFailures", "4") \
    # Максимальное количество попыток запуска стадии
    .config("spark.stage.maxConsecutiveAttempts", "4") \
    # Включение автоматического восстановления исполнителя
    .config("spark.yarn.max.executor.failures", "3") \
    # Режим спекулятивного выполнения (запуск копий медленных задач)
    .config("spark.speculation", "true") \
    .getOrCreate()
```

**Параметры специфичные для PySpark**:

```python
spark = SparkSession.builder \
    .appName("MySparkAppOnYARN") \
    .master("yarn") \
    # Максимальное количество строк, передаваемых в Python-процесс за раз
    .config("spark.python.worker.reuse", "true") \
    # Память для Python-процесса
    .config("spark.python.worker.memory", "512m") \
    # Максимальный размер сериализованного результата Python UDF
    .config("spark.python.worker.reuse.maxRetries", "3") \
    .getOrCreate()
```

**Практические рекомендации**:

1. **Для оптимизации ресурсов**:
   - Используйте динамическое выделение ресурсов (`spark.dynamicAllocation.enabled=true`)
   - Размер исполнителя (executor) выбирайте кратным размеру контейнера YARN
   - Помните о накладных расходах памяти (memoryOverhead)

2. **Для управления зависимостями**:
   - Используйте `spark.yarn.dist.archives` для Python библиотек
   - Используйте `spark.yarn.dist.jars` для Java/Scala библиотек
   - Для сложных зависимостей создавайте Conda/venv окружения и загружайте их как архивы

3. **Для мониторинга и отладки**:
   - Включите логирование событий (`spark.eventLog.enabled=true`)
   - Настройте уровень логирования в зависимости от потребностей
   - Используйте SparkUI или History Server для анализа

## Как можно выгружать логи в yarn и что они покажут?

Логи в YARN содержат ценную информацию о выполнении приложений, включая Spark-задачи. Их правильное использование помогает отлаживать и оптимизировать приложения.

**Способы выгрузки логов из YARN**:

1. **Командная строка (CLI)**:
   ```bash
   # Просмотр логов определенного приложения
   yarn logs -applicationId application_1234567890_0001
   
   # Выгрузка логов всех контейнеров приложения
   yarn logs -applicationId application_1234567890_0001 -am ALL > application_logs.txt
   
   # Выгрузка логов конкретного контейнера
   yarn logs -applicationId application_1234567890_0001 -containerId container_1234567890_0001_01_000001
   
   # Фильтрация по типу логов
   yarn logs -applicationId application_1234567890_0001 -log_files stderr
   ```

2. **Веб-интерфейс YARN**:
   - Перейдите в интерфейс ResourceManager (обычно http://yarn-rm-host:8088)
   - Найдите нужное приложение в списке
   - Нажмите на ID приложения для просмотра деталей
   - Перейдите во вкладку "Logs" для конкретного контейнера

3. **Программный доступ**:
   ```python
   # Через REST API YARN
   import requests
   import json
   
   app_id = "application_1234567890_0001"
   rm_url = "http://yarn-rm-host:8088"
   
   # Получение информации о приложении
   response = requests.get(f"{rm_url}/ws/v1/cluster/apps/{app_id}")
   app_info = json.loads(response.text)
   
   # Получение логов через WebHDFS API
   # (требует дополнительной настройки)
   ```

4. **Агрегация логов**:
   - YARN может агрегировать логи в HDFS для долгосрочного хранения
   - Настраивается в yarn-site.xml:
     ```xml
     <property>
       <name>yarn.log-aggregation-enable</name>
       <value>true</value>
     </property>
     <property>
       <name>yarn.nodemanager.log-dirs</name>
       <value>/var/log/hadoop-yarn/containers</value>
     </property>
     <property>
       <name>yarn.nodemanager.remote-app-log-dir</name>
       <value>/tmp/logs</value>
     </property>
     ```
   - Просмотр агрегированных логов:
     ```bash
     yarn logs -applicationId application_1234567890_0001 -appOwner username
     ```

**Что показывают логи YARN**:

1. **Логи ApplicationMaster**:
   - Процесс запуска и инициализации приложения
   - Запросы на выделение ресурсов
   - Статусы запуска контейнеров
   - Ошибки управления приложением
   - В Spark это логи SparkContext и драйвера

2. **Логи контейнеров**:
   - Для каждого контейнера создаются стандартные выводы:
     - `stdout` — стандартный вывод
     - `stderr` — стандартный вывод ошибок
     - `syslog` — системные сообщения контейнера
   - В Spark контейнеры содержат логи исполнителей (executors)

3. **Детализация по компонентам**:
   - **ResourceManager логи**:
     - Запросы на выделение ресурсов
     - Действия планировщика
     - Регистрация/отключение узлов
   
   - **NodeManager логи**:
     - Запуск/остановка контейнеров
     - Использование ресурсов
     - Ошибки на уровне узла
   
   - **Spark конкретные логи**:
     - Информация о стадиях и задачах
     - Метрики выполнения (время, объем данных)
     - Java/Python исключения
     - GC события и memory management

**Типичные сценарии использования логов**:

1. **Отладка сбоев приложения**:
   ```bash
   # Поиск ошибок в логах
   yarn logs -applicationId application_1234567890_0001 | grep -i "error\|exception\|failed"
   
   # Просмотр последних событий перед сбоем
   yarn logs -applicationId application_1234567890_0001 -containerId container_1234567890_0001_01_000002 | tail -n 100
   ```

2. **Анализ производительности**:
   - Поиск узких мест в логах исполнителей
   - Отслеживание длительных операций
   - Анализ использования ресурсов

3. **Мониторинг ресурсов**:

## Что можно увидеть в spark history?

**Spark History Server** — это веб-интерфейс для просмотра информации о завершенных и работающих Spark-приложениях. Он предоставляет детальную информацию о выполнении задач, которая помогает анализировать и оптимизировать Spark-приложения.

**Основная информация в Spark History Server**:

1. **Список приложений**:
   - ID приложения и его имя
   - Время запуска и продолжительность работы
   - Статус (завершено, работает, завершено с ошибкой)
   - Пользователь, запустивший приложение
   - Используемая версия Spark

2. **Детализация по приложению**:
   - **Jobs (Задания)** — высокоуровневые операции, вызванные действиями (actions)
     - Статус каждого задания (успешно, в процессе, с ошибкой)
     - Время выполнения и количество стадий
     - Успешные/неуспешные задачи

   - **Stages (Стадии)** — группы задач, образующие одно задание
     - Подробная информация о каждой стадии
     - Метрики выполнения (чтение/запись данных, shuffle)
     - Визуализация DAG (направленного ациклического графа)
     - Распределение задач по времени выполнения (гистограмма)

   - **Tasks (Задачи)** — отдельные единицы работы в стадии
     - Детали выполнения каждой задачи
     - Локальность данных (Data Locality)
     - Время выполнения, CPU-время, GC-время
     - Количество обработанных данных
     - Спекулятивное выполнение задач

   - **Storage (Хранилище)** — информация о кеширования RDD/DataFrame
     - Объем памяти, используемый для кеширования
     - Распределение кеша по исполнителям
     - Процент кешированных партиций

   - **Environment (Окружение)** — конфигурация Spark
     - Spark-свойства (spark.*)
     - Информация о системе
     - Версии компонентов
     - Runtime-информация (JVM, Scala, Java)

   - **Executors (Исполнители)** — рабочие процессы
     - Список всех исполнителей
     - Выделенная и использованная память
     - Дисковое пространство
     - Активные/завершенные задачи
     - GC-статистика

   - **SQL** — информация о SQL-запросах (если используется Spark SQL)
     - Текст запросов
     - Логический и физический планы выполнения
     - Метрики выполнения по каждому оператору

3. **Визуализации**:
   - **Event Timeline** — временная шкала событий приложения
     - Параллельное выполнение задач
     - Обнаружение узких мест и задержек
   
   - **DAG Visualization** — визуализация потока данных
     - Представление операций в виде графа
     - Зависимости между операциями

4. **Метрики и статистика**:
   - **Aggregated Metrics** — агрегированные метрики для каждой стадии
     - Среднее/максимальное время выполнения задач
     - Объемы считанных/записанных/переданных данных
   
   - **Shuffle Statistics** — статистика операций перемешивания
     - Чтение/запись по сети
     - Количество переданных записей

**Практическое применение Spark History Server**:

1. **Отладка и диагностика проблем**:
   - Анализ стека вызовов ошибок
   - Идентификация проблемных узлов или исполнителей
   - Выявление сбойных задач и причин их неудач

2. **Оптимизация производительности**:
   - Выявление медленных стадий и задач

   - Идентификация проблем с data skew (перекос данных)
   - Оптимизация параллелизма и планирования

3. **Мониторинг ресурсов**:

   - Мониторинг активности сборщика мусора (GC)

## Что такое DAG?

**DAG (Directed Acyclic Graph)** — это направленный ациклический граф, используемый в Spark и других системах для представления последовательности операций и зависимостей между ними.

**Ключевые характеристики DAG**:
- **Направленный** — все ребра имеют направление (от одной операции к другой)
- **Ациклический** — нет циклов, операции не могут зависеть от своих результатов
- **Граф** — структура, состоящая из узлов (операций) и ребер (зависимостей)

**DAG в контексте Spark**:

1. **Физическое представление программы**:
   - Каждое преобразование (transformation) и действие (action) становится узлом в графе
   - Зависимости между операциями определяют ребра графа
   - Spark автоматически создает DAG на основе кода приложения

2. **Типы зависимостей в DAG**:
   - **Узкие зависимости (Narrow Dependencies)** — каждая партиция родителя влияет только на одну партицию потомка
     - Примеры: map, filter, union
     - Могут выполняться как одна стадия (stage)
   
   - **Широкие зависимости (Wide Dependencies)** — одна партиция родителя влияет на несколько партиций потомка
     - Примеры: groupByKey, join, repartition
     - Требуют операции shuffle (перемешивания данных)
     - Создают границы между стадиями

3. **Оптимизация через DAG**:
   - Spark анализирует DAG для оптимизации выполнения
   - Объединяет последовательные узкие преобразования (pipelining)
   - Минимизирует количество операций shuffle
   - Применяет каталитические оптимизации для SQL-запросов

**DAG в контексте Airflow**:

1. **Представление рабочего процесса**:
   - Узлы графа — задачи (tasks)
   - Ребра — зависимости между задачами
   - Определяет порядок выполнения задач

2. **Планирование выполнения**:
   - Параллельное выполнение независимых задач
   - Задачи запускаются только после успешного выполнения всех предшествующих задач
   - Контроль потока выполнения рабочего процесса

**Визуализация DAG**:

В Spark UI и History Server DAG визуализируется для анализа:
- Узлы представляют операции
- Ребра показывают зависимости и потоки данных
- Цвета часто отражают статус или тип операции

**Практическое применение понимания DAG**:

1. **Оптимизация кода**:
   - Минимизация операций shuffle путем правильного порядка преобразований
   - Группировка операций для эффективного выполнения
   - Использование правильных операций для нужных сценариев

2. **Отладка и диагностика**:
   - Анализ DAG помогает понять план выполнения
   - Выявление сложных или неэффективных операций
   - Понимание причин сбоев или замедлений

3. **Планирование ресурсов**:
   - Оценка требуемых ресурсов на основе структуры DAG
   - Выявление узких мест и потенциальных проблем

**Пример DAG в Spark**:

```python
# Создание и выполнение DAG в Spark
rdd = spark.sparkContext.textFile("hdfs:///data/text.txt")  # Загрузка данных

words = rdd.flatMap(lambda line: line.split(" "))           # Узкая зависимость
pairs = words.map(lambda word: (word, 1))                   # Узкая зависимость
wordCounts = pairs.reduceByKey(lambda a, b: a + b)          # Широкая зависимость (shuffle)
sortedCounts = wordCounts.sortBy(lambda x: x[1], False)     # Широкая зависимость (shuffle)

# Действие, запускающее выполнение DAG
top10 = sortedCounts.take(10)
```

В этом примере Spark создаст DAG с двумя стадиями, разделенными операциями shuffle:
1. Загрузка файла, разбиение на слова, и создание пар (flatMap, map)
2. Подсчет слов (reduceByKey с shuffle)
3. Сортировка результатов (sortBy с shuffle)

## Какие основные параметры прописываются в AirFlow?

**Apache Airflow** — это платформа для программного создания, планирования и мониторинга рабочих процессов (workflow). Ниже приведены основные параметры, которые настраиваются в Airflow.

**1. Параметры конфигурации на уровне системы** (airflow.cfg):

- **Core Параметры**:
  - `dags_folder` — путь к директории с DAG-файлами
  - `default_timezone` — часовой пояс по умолчанию
  - `executor` — тип исполнителя (SequentialExecutor, LocalExecutor, CeleryExecutor, KubernetesExecutor)
  - `parallelism` — максимальное количество одновременно выполняемых задач
  - `dag_concurrency` — максимальное количество задач, выполняемых в одном DAG
  - `max_active_runs_per_dag` — максимальное количество одновременных запусков DAG

- **Параметры планирования**:
  - `scheduler_heartbeat_sec` — интервал проверки планировщиком новых задач
  - `min_file_process_interval` — минимальный интервал между сканированием DAG-файлов
  - `dag_dir_list_interval` — интервал проверки директории DAG на новые файлы
  - `catchup_by_default` — включить/выключить catchup по умолчанию

- **Параметры веб-сервера**:
  - `base_url` — базовый URL для веб-сервера
  - `web_server_port` — порт веб-сервера
  - `web_server_host` — хост веб-сервера
  - `secret_key` — секретный ключ для сессий и куки

- **Параметры базы данных**:
  - `sql_alchemy_conn` — строка подключения к базе данных
  - `sql_engine_encoding` — кодировка для базы данных
  - `sql_alchemy_pool_size` — размер пула соединений

- **Параметры логирования**:
  - `logging_level` — уровень логирования
  - `logging_config_class` — путь к кастомной конфигурации логирования
  - `base_log_folder` — директория для хранения логов

```python
# Назначение задачи в пул
task = BashOperator(
    task_id='resource_intensive_task',
    bash_command='heavy_script.sh',
    pool='my_custom_pool',
    pool_slots=2  # Задача займет 2 слота из пула
)
```

**7. XComs для обмена данными между задачами**:

```python
# Отправка значения в XCom
def push_function(**context):
    context['task_instance'].xcom_push(key='my_key', value='my_value')

# Получение значения из XCom
def pull_function(**context):
    value = context['task_instance'].xcom_pull(task_ids='push_task', key='my_key')
```

**Практические рекомендации**:

1. **Для производительности**:
   - Увеличьте `parallelism` и `dag_concurrency` для большего параллелизма
   - Используйте CeleryExecutor для распределенного выполнения
   - Настройте пулы для ограничения ресурсоемких задач

2. **Для надежности**:
   - Настройте параметры повторных попыток для задач
   - Используйте SLA для мониторинга длительности задач
   - Настройте обратные вызовы для уведомлений о проблемах

3. **Для масштабирования**:
   - Используйте внешнюю базу данных (PostgreSQL, MySQL)
   - Настройте распределенный кеш (Redis)
   - Применяйте субдаги для управления сложными рабочими процессами

## Что такое bash оператор в Airflow?

**Bash Operator** в Apache Airflow — это оператор, который выполняет команды оболочки Bash. Это один из самых простых и часто используемых операторов в Airflow, который позволяет запускать скрипты и команды на узле, где запущен исполнитель Airflow.

**Основные характеристики BashOperator**:

1. **Функциональность**:
   - Выполняет Bash-команды или скрипты
   - Позволяет определить переменные окружения
   - Может использовать шаблоны Jinja2 для динамической генерации команд
   - Возвращает код выхода команды (0 означает успех, другие значения — ошибку)

2. **Параметры**:
   - `bash_command` (обязательный) — команда или скрипт для выполнения
   - `env` — словарь переменных окружения
   - `output_encoding` — кодировка вывода команды
   - `cwd` — рабочая директория для выполнения команды

3. **Использование Jinja-шаблонов**:
   - Можно использовать переменные контекста в команде
   - Доступ к параметрам времени выполнения
   - Интерполяция значений из других задач или Airflow Variables

**Пример использования BashOperator**:

```python
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

default_args = {
    'owner': 'airflow',
    'depends_on_past': False,
    'start_date': datetime(2023, 1, 1),
    'retries': 1,
    'retry_delay': timedelta(minutes=5),
}

dag = DAG(
    'bash_example',
    default_args=default_args,
    description='Example DAG using BashOperator',
    schedule_interval='@daily',
)

# Простая команда
task1 = BashOperator(
    task_id='print_date',
    bash_command='date',
    dag=dag,
)

# Команда с использованием переменных окружения
task2 = BashOperator(
    task_id='echo_env',
    bash_command='echo $MY_ENV_VAR',
    env={'MY_ENV_VAR': 'Hello from Airflow!'},
    dag=dag,
)

# Использование Jinja-шаблонов
task3 = BashOperator(
    task_id='templated_command',
    bash_command='echo "Execution date: {{ ds }} with run_id {{ run_id }}"',
    dag=dag,
)

# Выполнение скрипта
task4 = BashOperator(
    task_id='run_script',
    bash_command='bash /path/to/my_script.sh {{ params.input_file }}',
    params={'input_file': '/data/input.csv'},
    dag=dag,
)

# Пример обработки файла с использованием переменной execution_date
task5 = BashOperator(
    task_id='process_data',
    bash_command='python /scripts/process.py --date {{ ds }} --output /data/output_{{ ds_nodash }}.csv',
    dag=dag,
)

# Определение порядка выполнения
task1 >> task2 >> task3 >> [task4, task5]
```

**Практические советы по использованию BashOperator**:

1. **Обработка ошибок**:
   - BashOperator считает задачу неудачной, если код выхода не равен 0
   - Используйте конструкции Bash для обработки ошибок:
     ```bash
     set -e  # Завершение при ошибке
     command || { echo "Command failed"; exit 1; }
     ```

2. **Безопасное использование переменных**:
   - Экранируйте переменные для безопасного использования:
     ```python
     bash_command='echo "{{ params.value | quote }}"'
     ```

3. **Передача данных между задачами**:
   - Сохранение вывода команды в XCom:
     ```python
     task = BashOperator(
         task_id='save_output',
         bash_command='echo "Value to be shared"',
         do_xcom_push=True,  # Автоматически сохраняет вывод в XCom
         dag=dag,
     )
     ```

4. **Управление временем выполнения**:
   - Установка таймаута для долго выполняющихся команд:
     ```python
     task = BashOperator(
         task_id='long_running_task',
         bash_command='sleep 3600',  # 1 час
         execution_timeout=timedelta(minutes=30),  # Таймаут 30 минут
         dag=dag,
     )
     ```

5. **Использование многострочных команд**:
   ```python
   task = BashOperator(
       task_id='multiline_script',
       bash_command='''
          echo "Starting script"
          cd /path/to/workdir
          python process.py
          if [ $? -eq 0 ]; then
             echo "Success"
          else
             echo "Failed"
             exit 1
          fi
       ''',
       dag=dag,
   )
   ```

**Преимущества и недостатки BashOperator**:

**Преимущества**:
- Простота использования для быстрой автоматизации
- Не требует написания Python-кода для простых операций
- Использует знакомые bash-команды и утилиты UNIX
- Легко запускать существующие скрипты

**Недостатки**:
- Ограниченная возможность отладки
- Зависит от окружения, где выполняется оператор
- Не обеспечивает переносимость между разными ОС
- Может создавать сложности при обработке сложных структур данных

**Когда использовать BashOperator**:
- Для выполнения простых команд оболочки
- Для запуска существующих скриптов
- Для операций с файловой системой
- Когда не требуется сложная логика, которая лучше реализуется в Python

## Что такое файл requirements.txt и как его в Airflow подкладывать?

**Requirements.txt** — это файл, который содержит список Python-пакетов, необходимых для работы проекта. В контексте Airflow, этот файл используется для установки дополнительных Python-зависимостей, необходимых для выполнения DAG и задач.

**Структура файла requirements.txt**:
- Каждая строка содержит одно имя пакета
- Можно указать конкретную версию пакета
- Поддерживает синтаксис pip для указания версий

Пример содержимого requirements.txt:
```
pandas==1.3.5
numpy>=1.20.0
scikit-learn
apache-airflow-providers-postgres
requests>=2.25.0,<3.0.0
```

**Способы использования requirements.txt в Airflow**:

1. **Установка зависимостей в образ Airflow**:
   - При использовании Docker, включите requirements.txt в образ:
     ```dockerfile
     FROM apache/airflow:2.3.0
     COPY requirements.txt /requirements.txt
     RUN pip install --user --no-cache-dir -r /requirements.txt
     ```

2. **Использование PythonVirtualenvOperator**:
   - Создает виртуальное окружение и устанавливает зависимости для каждой задачи:
     ```python
     from airflow.operators.python import PythonVirtualenvOperator
     
     def my_function():
         import pandas as pd  # pandas будет установлен во временное виртуальное окружение
         # код задачи
     
     task = PythonVirtualenvOperator(
         task_id='virtualenv_task',
         python_callable=my_function,
         requirements=['pandas==1.3.5', 'numpy>=1.20.0'],  # или requirements='/path/to/requirements.txt',
         system_site_packages=False,
         dag=dag,
     )
     ```

3. **Использование KubernetesPodOperator**:
   - При запуске Airflow в Kubernetes, можно указать зависимости для контейнера:
     ```python
     from airflow.providers.cncf.kubernetes.operators.kubernetes_pod import KubernetesPodOperator
     
     task = KubernetesPodOperator(
         task_id='kubernetes_pod_task',
         name='task-with-deps',
         namespace='default',
         image='python:3.9',
         cmds=["bash", "-c"],
         arguments=["pip install -r /opt/requirements.txt && python /opt/script.py"],
         # Монтирование файлов в контейнер
         volumes=[volume],
         volume_mounts=[volume_mount],
         dag=dag,
     )
     ```

4. **Использование DockerOperator**:
   - Запуск задачи в Docker-контейнере с установленными зависимостями:
     ```python
     from airflow.providers.docker.operators.docker import DockerOperator
     
     task = DockerOperator(
         task_id='docker_task',
         image='python:3.9',
         command='bash -c "pip install -r /tmp/requirements.txt && python /tmp/script.py"',
         volumes=['/local/path/to/requirements.txt:/tmp/requirements.txt',
                  '/local/path/to/script.py:/tmp/script.py'],
         dag=dag,
     )
     ```

5. **Для пользовательских плагинов**:
   - Создание плагина с собственными операторами, хуками или сенсорами:
     ```
     airflow_plugins/
     ├── __init__.py
     ├── my_plugin.py
     └── requirements.txt
     ```
   - Необходимо установить зависимости в окружение Airflow:
     ```bash
     pip install -r airflow_plugins/requirements.txt
     ```

**Практические рекомендации**:

1. **Управление конфликтами зависимостей**:
   - Будьте осторожны с версиями пакетов, которые могут конфликтовать с основными зависимостями Airflow
   - Используйте виртуальные окружения для изоляции зависимостей
   - Тестируйте зависимости перед развертыванием в production

2. **Оптимизация установки зависимостей**:
   - Минимизируйте список необходимых пакетов
   - Указывайте конкретные версии для предсказуемости
   - Рассмотрите возможность предварительной сборки образов с необходимыми зависимостями

3. **Безопасность**:
   - Проверяйте зависимости на известные уязвимости
   - Используйте доверенные источники пакетов
   - Рассмотрите возможность использования закрытых репозиториев пакетов

**Примеры для разных сценариев**:

1. **Для простого DAG с дополнительными зависимостями**:
   - Создайте requirements.txt в директории с DAG:
     ```
     # requirements.txt для DAG1
     ```

## Какие зависимости в Airflow можно протягивать?

В Airflow можно определять различные типы зависимостей между задачами для управления потоком выполнения рабочего процесса. Эти зависимости определяют порядок и условия выполнения задач.

**Основные типы зависимостей в Airflow**:

1. **Стандартные зависимости (Standard Dependencies)**:
   - Определяют последовательность выполнения задач
   - Задача запускается, только когда все задачи, от которых она зависит, успешно выполнились
   - Способы определения:
     ```python
     # Использование операторов-битшифт (рекомендуемый способ)
     task1 >> task2 >> task3
     
     # Эквивалентно
     task1.set_downstream(task2)
     task2.set_downstream(task3)
     
     # Или обратная зависимость
     task3 << task2 << task1
     
     # Эквивалентно
     task3.set_upstream(task2)
     task2.set_upstream(task1)
     ```

2. **Зависимости от нескольких задач (Multiple Dependencies)**:
   - Определяют зависимость от нескольких задач одновременно
   - Примеры:
     ```python
     # Задача зависит от нескольких предыдущих
     [task1, task2, task3] >> task4
     
     # Несколько задач зависят от одной предыдущей
     task1 >> [task2, task3, task4]
     
     # Параллельные ветви объединяются
     task1 >> [task2, task3] >> task4
     
     # Сложные комбинации
     task1 >> [task2, task3]
     task3 >> task4
     [task2, task4] >> task5
     ```

3. **Триггерные правила (Trigger Rules)**:
   - Определяют условия запуска задачи в зависимости от статуса предшествующих задач
   - Основные правила:
     - `all_success` (по умолчанию): запуск, если все предшествующие задачи успешны
     - `all_failed`: запуск, если все предшествующие задачи завершились неудачно
     - `all_done`: запуск после завершения всех предшествующих задач (успешно или неудачно)
     - `one_failed`: запуск, если хотя бы одна предшествующая задача завершилась неудачно
     - `one_success`: запуск, если хотя бы одна предшествующая задача успешна
     - `none_failed`: запуск, если все предшествующие задачи не завершились неудачно
     - `none_skipped`: запуск, если ни одна из предшествующих задач не была пропущена
   - Пример использования:
     ```python
     task = BashOperator(
         task_id='error_handler',
         bash_command='./error_handling.sh',
         trigger_rule='one_failed',
         dag=dag,
     )
     ```

4. **Условные ветвления (Branching)**:
   - Определяют динамические пути выполнения на основе результатов предыдущих задач
   - Используется оператор `BranchPythonOperator`:
     ```python
     def branch_func(**context):
         # Логика определения следующей задачи
         if condition:
             return 'task_a'
         else:
             return 'task_b'
     
     branching = BranchPythonOperator(
         task_id='branching',
         python_callable=branch_func,
         dag=dag,
     )
     
     task_a = BashOperator(...)
     task_b = BashOperator(...)
     
     branching >> [task_a, task_b]
     ```

5. **Зависимости от выполнения предыдущего экземпляра задачи (Depends On Past)**:
   - Задача зависит от успешного выполнения своего предыдущего экземпляра
   - Полезно для обеспечения последовательной обработки инкрементальных данных
   - Пример:
     ```python
     task = BashOperator(
         task_id='incremental_processing',
         bash_command='./process_data.sh',
         depends_on_past=True,
         dag=dag,
     )
     ```

6. **Ожидание выполнения нижестоящих задач (Wait For Downstream)**:
   - Задача не начнет следующий запуск, пока все нижестоящие задачи не завершат текущий запуск
   - Помогает избежать конфликтов между запусками
   - Пример:
     ```python
     task = BashOperator(
         task_id='parent_task',
         bash_command='./parent_process.sh',
         wait_for_downstream=True,
         dag=dag,
     )
     ```

7. **Внешние зависимости (Cross-DAG Dependencies)**:
   - Зависимости между задачами из разных DAG
   - Используется оператор `ExternalTaskSensor`:
     ```python
     # В DAG2 ожидается завершение задачи из DAG1
     wait_for_dag1 = ExternalTaskSensor(
         task_id='wait_for_dag1',
         external_dag_id='dag1',
         external_task_id='final_task',
         execution_delta=timedelta(hours=1),  # Временная дельта между выполнениями DAG
         dag=dag2,
     )
     ```

**Продвинутые типы зависимостей**:

1. **Динамические зависимости**:
   - Зависимости, создаваемые во время выполнения
   - Могут быть основаны на результатах предыдущих задач или внешних условиях
   - Пример с TaskFlow API:
     ```python
     @task
     def get_tasks():
         # Динамически определяем список задач
         return ['task_a', 'task_b', 'task_c']
     
     @task
     def process(task_name):
         # Обработка для каждой задачи
         print(f"Processing {task_name}")
     
     tasks = get_tasks()
     processed = process.expand(task_name=tasks)
     ```

2. **Зависимости, основанные на данных**:
   - Зависимости, которые определяются на основе данных, передаваемых между задачами
   - Используют XComs для обмена данными
   - Пример:
     ```python
     def check_data(**context):
         data_quality = compute_quality_score()
         if data_quality > 0.9:
             return 'high_quality_path'
         else:
             return 'low_quality_path'
     
     data_branch = BranchPythonOperator(
         task_id='check_data_quality',
         python_callable=check_data,
         dag=dag,
     )
     ```

3. **Временные зависимости**:
   - Зависимости, основанные на времени
   - Используются сенсоры (TimeSensor, TimeDeltaSensor) для ожидания определенного времени
   - Пример:
     ```python
     wait_until_10am = TimeSensor(
         task_id='wait_until_10am',
         target_time='10:00:00',
         dag=dag,
     )
     ```

**Практические рекомендации**:

1. **Проектирование DAG**:
   - Разделяйте сложные DAG на логические группы задач
   - Используйте SubDAG или TaskGroups для организации сложных рабочих процессов
   - Избегайте циклических зависимостей

2. **Обработка ошибок**:
   - Используйте `trigger_rule` для определения поведения при ошибках
   - Создавайте обработчики ошибок с использованием зависимостей
   - Настраивайте уведомления для критических задач

3. **Оптимизация выполнения**:
   - Проектируйте DAG для максимального параллелизма
   - Минимизируйте ненужные зависимости
   - Используйте пулы для контроля параллельного выполнения ресурсоемких задач

4. **Тестирование зависимостей**:
   - Используйте `airflow test` для проверки задач и их зависимостей
   - Создавайте unit-тесты для логики определения зависимостей
   - Визуализируйте DAG в UI для проверки структуры зависимостей

## Какое время в Airflow выстраивается?

**Время в Airflow** — это важная концепция, которая определяет, когда и как выполняются задачи. Airflow предоставляет гибкую систему для работы с временем, которая может быть адаптирована под различные сценарии использования.

**Основные концепции времени в Airflow**:

1. **Execution Date (дата выполнения)**:
   - Не фактическое время запуска, а логическое время, к которому относится запуск DAG
   - Обычно представляет собой начало периода, за который выполняется обработка
   - В Airflow 2.0+ переименована в logical_date, но концепция остается той же
   - Доступна в шаблонах как `{{ ds }}` (формат YYYY-MM-DD) или `{{ execution_date }}`

2. **Schedule Interval (интервал планирования)**:
   - Определяет, как часто должен запускаться DAG
   - Может быть определен с помощью:
     - Cron-выражения: '0 0 * * *' (ежедневно в полночь)
     - Presets: @daily, @weekly, @monthly, @hourly
     - timedelta объекта: timedelta(days=1)
   - DAG запускается после окончания интервала планирования

3. **Start Date (дата начала)**:
   - Время, с которого Airflow начинает планировать выполнение DAG
   - Задается при создании DAG или в default_args
   - Важна для определения первого запуска и расчета отсутствующих запусков (catchup)

4. **End Date (дата окончания)**:
   - Время, после которого Airflow прекращает планировать выполнение DAG
   - Необязательный параметр, по умолчанию DAG планируется бесконечно

**Как Airflow планирует выполнение DAG**:

1. **Расчет времени запуска**:
   - Airflow создает DagRun после окончания интервала планирования
   - Execution date = start_date + N * schedule_interval
   - Фактическое время запуска (run_date) = execution_date + schedule_interval

2. **Пример расчета времени**:
   - DAG с start_date='2023-01-01 00:00:00' и schedule_interval='@daily'
   - Первый запуск будет иметь execution_date='2023-01-01 00:00:00'
   - Фактически запуск произойдет после окончания интервала: '2023-01-02 00:00:00'
   - Этот запуск обрабатывает данные за день 2023-01-01

3. **Наглядное представление**:
   ```
   Execution Date | Schedule Interval | Next Execution Date | Runtime
   --------------------------------------------------------------
   2023-01-01     | @daily (24h)      | 2023-01-02          | 2023-01-02
   2023-01-02     | @daily (24h)      | 2023-01-03          | 2023-01-03
   2023-01-03     | @daily (24h)      | 2023-01-04          | 2023-01-04
   ```

**Часовые пояса в Airflow**:

1. **Настройка часового пояса**:
   - Глобальная настройка в airflow.cfg:
     ```
     default_timezone = UTC  # или другой часовой пояс, например Europe/Moscow
     ```
   - На уровне DAG:
     ```python
     dag = DAG(
         'timezone_example',
         default_args=default_args,
         schedule_interval='0 9 * * *',  # 9:00 утра в указанном часовом поясе
         start_date=datetime(2023, 1, 1, tzinfo=timezone('Europe/Moscow')),
     )
     ```

2. **Интерпретация времени**:
   - Все времена хранятся в UTC
   - Отображение в UI в настроенном часовом поясе
   - Cron-выражения интерпретируются в указанном часовом поясе
   - При использовании разных часовых поясов нужно быть особенно внимательным к DST (летнее/зимнее время)

**Дополнительные концепции времени**:

1. **Catchup (наверстывание)**:
   - Determines if Airflow should schedule runs to catch up from the start_date
   - Если DAG имеет start_date в прошлом и catchup=True, Airflow создаст запуски для всех пропущенных интервалов
   - Можно отключить глобально или на уровне DAG:
     ```python
     dag = DAG(
         'no_catchup_example',
         default_args=default_args,
         schedule_interval='@daily',
         catchup=False,
     )
     ```

2. **Backfill (обратное заполнение)**:
   - Ручной запуск DAG для исторических интервалов времени
   - Полезно для переобработки данных или тестирования:
     ```bash
     airflow dags backfill my_dag -s 2023-01-01 -e 2023-01-10
     ```

3. **Время для задач с датами**:
   - В шаблонах доступны различные форматы времени:
     - `{{ ds }}` - Execution date в формате YYYY-MM-DD
     - `{{ ds_nodash }}` - Execution date в формате YYYYMMDD
     - `{{ prev_ds }}` - Предыдущая execution date в формате YYYY-MM-DD
     - `{{ next_ds }}` - Следующая execution date в формате YYYY-MM-DD
     - `{{ execution_date }}` - Полный объект datetime
   - Пример использования в задаче:
     ```python
     task = BashOperator(
         task_id='date_task',
         bash_command='echo "Processing data for {{ ds }}"',
         dag=dag,
     )
     ```

**Практические рекомендации**:

1. **Выбор правильного интервала планирования**:
   - Учитывайте доступность данных и бизнес-требования
   - Для ETL-процессов часто используются ежедневные или ежечасные интервалы
   - Для отчетов может подойти еженедельный или ежемесячный интервал

2. **Работа с датами в задачах**:
   - Используйте переменные шаблонов для передачи дат в задачи
   - Для сложных расчетов дат используйте PythonOperator:
     ```python
     def calculate_dates(**context):
         execution_date = context['execution_date']
         # Расчеты с датами
         return {'start_date': start_date.strftime('%Y-%m-%d'), 'end_date': end_date.strftime('%Y-%m-%d')}
     
     date_task = PythonOperator(
         task_id='calculate_dates',
         python_callable=calculate_dates,
         provide_context=True,
         dag=dag,
     )
     ```

3. **Обработка краевых случаев**:
   - Учитывайте переходы на летнее/зимнее время при работе с временными рядами
   - Будьте внимательны к високосным годам и разному количеству дней в месяцах
   - Для критичных ко времени задач рассмотрите использование сенсоров:
     ```python
     time_sensor = TimeSensor(
         task_id='wait_until_business_hours',
         target_time='{{ execution_date.strftime("%H:%M") }}',
         dag=dag,
     )
     ```

## Что такое XCOM?

**XCom (Cross-Communication)** в Airflow — это механизм для обмена небольшими объемами данных между задачами в рамках одного DAG или даже между разными DAG-ами. XCom позволяет передавать результаты одной задачи другим задачам, обеспечивая коммуникацию и обмен состоянием.

**Основные характеристики XCom**:

1. **Назначение**:
   - Передача данных между задачами
   - Сохранение результатов выполнения задач
   - Обмен контекстной информацией в DAG

2. **Хранение**:
   - XCom хранятся в базе данных Airflow
   - Предназначены для небольших объемов данных (обычно до нескольких МБ)
   - Имеют ограничения на размер в зависимости от используемой БД

3. **Структура XCom**:
   - `key` — имя, под которым сохраняется значение
   - `value` — сохраняемые данные (подлежат сериализации)
   - `task_id` — ID задачи, создавшей XCom
   - `dag_id` — ID DAG, в котором была создана XCom
   - `execution_date` — дата выполнения DAG, к которой относится XCom

**Способы работы с XCom**:

1. **Явная отправка (push) значения в XCom**:
   ```python
   def push_xcom_value(**context):
       # Отправка значения в XCom
       context['task_instance'].xcom_push(key='sample_key', value='sample_value')
       # Можно отправить сложные объекты (словари, списки), они будут сериализованы
       context['task_instance'].xcom_push(key='result_data', value={'status': 'success', 'count': 42})
   
   push_task = PythonOperator(
       task_id='push_xcom_example',
       python_callable=push_xcom_value,
       provide_context=True,
       dag=dag,
   )
   ```

2. **Получение (pull) значения из XCom**:
   ```python
   def pull_xcom_value(**context):
       # Получение значения по ключу и ID задачи
       value = context['task_instance'].xcom_pull(task_ids='push_xcom_example', key='sample_key')
       print(f"Retrieved value: {value}")
       
       # Получение всех XCom от задачи (без указания key)
       all_values = context['task_instance'].xcom_pull(task_ids='push_xcom_example')
       print(f"All values: {all_values}")
   
   pull_task = PythonOperator(
       task_id='pull_xcom_example',
       python_callable=pull_xcom_value,
       provide_context=True,
       dag=dag,
   )
   
   # Определение зависимости (pull_task выполняется после push_task)
   push_task >> pull_task
   ```

3. **Автоматическая отправка возвращаемого значения**:
   ```python
   # Значение, возвращаемое функцией, автоматически отправляется в XCom
   # с ключом 'return_value'
   def auto_push_xcom(**context):
       return "This will be automatically pushed to XCom"
   
   auto_task = PythonOperator(
       task_id='auto_push_example',
       python_callable=auto_push_xcom,
       provide_context=True,
       dag=dag,
   )
   
   def use_auto_value(**context):
       # Получение автоматически отправленного значения
       value = context['task_instance'].xcom_pull(task_ids='auto_push_example')
       # или с явным указанием ключа:
       # value = context['task_instance'].xcom_pull(task_ids='auto_push_example', key='return_value')
       print(f"Auto-pushed value: {value}")
   
   use_task = PythonOperator(
       task_id='use_auto_example',
       python_callable=use_auto_value,
       provide_context=True,
       dag=dag,
   )
   
   auto_task >> use_task
   ```

**Практические примеры использования XCom**:

1. **Динамическое ветвление на основе результатов задачи**:
   ```python
   def check_data_quality(**context):
       # Анализ качества данных
       quality_score = analyze_data()
       # Возвращаем результат для использования в BranchPythonOperator
       return quality_score
   
   quality_task = PythonOperator(
       task_id='check_quality',
       python_callable=check_data_quality,
       dag=dag,
   )
   
   def decide_path(**context):
       # Получение результата проверки качества
       quality_score = context['task_instance'].xcom_pull(task_ids='check_quality')
       if quality_score > 0.9:
           return 'high_quality_path'
       else:
           return 'low_quality_path'
   
   branch_task = BranchPythonOperator(
       task_id='branch_on_quality',
       python_callable=decide_path,
       dag=dag,
   )
   
   high_quality = DummyOperator(task_id='high_quality_path', dag=dag)
   low_quality = DummyOperator(task_id='low_quality_path', dag=dag)
   
   quality_task >> branch_task >> [high_quality, low_quality]
   ```

2. **Агрегация результатов параллельных задач**:
   ```python
   def process_partition(partition_id, **context):
       # Обработка части данных
       result = process_data_chunk(partition_id)
       return {'partition_id': partition_id, 'count': len(result), 'sum': sum(result)}
   
   # Создание нескольких задач для параллельной обработки
   partition_tasks = []
   for i in range(5):
       task = PythonOperator(
           task_id=f'process_partition_{i}',
           python_callable=process_partition,
           op_kwargs={'partition_id': i},
           dag=dag,
       )
       partition_tasks.append(task)
   
   def aggregate_results(**context):
       # Сбор результатов всех партиций
       results = []
       for i in range(5):
           partition_result = context['task_instance'].xcom_pull(task_ids=f'process_partition_{i}')
           results.append(partition_result)
       
       # Агрегация результатов
       total_count = sum(r['count'] for r in results)
       total_sum = sum(r['sum'] for r in results)
       average = total_sum / total_count if total_count > 0 else 0
       
       print(f"Total count: {total_count}, Average: {average}")
       return {'total_count': total_count, 'total_sum': total_sum, 'average': average}
   
   aggregate_task = PythonOperator(
       task_id='aggregate_results',
       python_callable=aggregate_results,
       dag=dag,
   )
   
   partition_tasks >> aggregate_task
   ```

**Ограничения и рекомендации**:

1. **Размер данных**:
   - XCom предназначены для небольших объемов данных
   - Большие данные (более нескольких МБ) лучше передавать через внешние хранилища (S3, HDFS, базы данных)
   - Пример с большими данными:
     ```python
     def process_large_data(**context):
         # Вместо передачи больших данных через XCom
         result_data = process_function()
         
         # Сохраняем результат во внешнем хранилище
         output_path = f"s3://bucket/results/{context['ds']}/result.csv"
         save_to_s3(result_data, output_path)
         
         # Через XCom передаем только путь к данным
         return {'result_path': output_path}
     ```

2. **Сериализация**:
   - Данные должны быть сериализуемыми
   - По умолчанию используется pickle, что может вызывать проблемы совместимости
   - Рекомендуется использовать JSON-сериализуемые типы для совместимости

3. **Управление жизненным циклом**:
   - XCom хранятся в базе данных Airflow бессрочно, если не настроена очистка
   - Для больших кластеров рекомендуется настроить периодическую очистку старых XCom

## Что такое Connections?

**Connections** в Airflow — это способ хранения и управления информацией о подключениях к внешним системам и сервисам. Они обеспечивают централизованное хранение учетных данных и параметров подключения, которые можно безопасно использовать в задачах Airflow.

**Основные характеристики Connections**:

1. **Назначение**:
   - Хранение учетных данных (логин, пароль, токены)
   - Хранение параметров подключения (хост, порт, схема)
   - Обеспечение безопасного доступа к внешним системам

2. **Структура Connection**:
   - `conn_id` — уникальный идентификатор подключения
   - `conn_type` — тип подключения (postgres, mysql, http, aws, etc.)
   - `host` — хост или URL сервера
   - `schema` — схема базы данных или дополнительная информация
   - `login` — имя пользователя
   - `password` — пароль (хранится в зашифрованном виде)
   - `port` — порт для подключения
   - `extra` — дополнительные параметры в формате JSON

**Способы управления Connections**:

1. **Через веб-интерфейс Airflow**:
   - Admin → Connections
   - Создание, редактирование, удаление подключений
   - Доступ к истории изменений

2. **Через командную строку (CLI)**:
   ```bash
   # Добавление нового подключения
   airflow connections add \
       --conn-id 'my_postgres_conn' \
       --conn-type 'postgres' \
       --conn-host 'localhost' \
       --conn-login 'airflow' \
       --conn-password 'airflow' \
       --conn-schema 'airflow' \
       --conn-port '5432'
   
   # Список всех подключений
   airflow connections list
   
   # Получение информации о конкретном подключении
   airflow connections get my_postgres_conn
   
   # Удаление подключения
   airflow connections delete my_postgres_conn
   ```

3. **Через переменные окружения**:
   ```bash
   # Формат: AIRFLOW_CONN_<CONN_ID_UPPERCASE>
   export AIRFLOW_CONN_MY_POSTGRES_CONN='postgres://airflow:airflow@localhost:5432/airflow'
   
   # Или с использованием JSON для extra
   export AIRFLOW_CONN_MY_POSTGRES_CONN='{"conn_type": "postgres", "host": "localhost", "login": "airflow", "password": "airflow", "schema": "airflow", "port": 5432, "extra": {"param1": "value1"}}'
   ```

**Использование Connections в задачах**:

1. **Через Hooks (рекомендуемый способ)**:
   ```python
   from airflow.hooks.postgres_hook import PostgresHook
   
   def query_postgres(**context):
       # Использование подключения через Hook
       pg_hook = PostgresHook(postgres_conn_id='my_postgres_conn')
       
       # Выполнение запросов
       connection = pg_hook.get_conn()
       cursor = connection.cursor()
       cursor.execute("SELECT COUNT(*) FROM users")
       count = cursor.fetchone()[0]
       
       print(f"Total users: {count}")
       return count
   
   postgres_task = PythonOperator(
       task_id='query_postgres',
       python_callable=query_postgres,
       dag=dag,
   )
   ```

2. **Получение информации о подключении напрямую**:
   ```python
   from airflow.hooks.base_hook import BaseHook
   
   def use_connection_directly(**context):
       # Получение информации о подключении
       conn = BaseHook.get_connection('my_postgres_conn')
       
       # Использование атрибутов подключения
       host = conn.host
       user = conn.login
       password = conn.password
       database = conn.schema
       port = conn.port
       
       # Получение дополнительных параметров
       extra_dict = conn.extra_dejson
       param1 = extra_dict.get('param1')
       
       print(f"Connecting to {host}:{port}/{database} as {user}")
       # Далее код подключения с использованием полученных параметров
   ```

**Типы подключений и соответствующие Hooks**:

Airflow предоставляет множество встроенных Hooks для различных типов подключений:

1. **Базы данных**:
   - `postgres_hook.PostgresHook` — для PostgreSQL
   - `mysql_hook.MySqlHook` — для MySQL
   - `mssql_hook.MsSqlHook` — для Microsoft SQL Server
   - `oracle_hook.OracleHook` — для Oracle

2. **Облачные сервисы**:
   - `aws_hook.AwsHook` — базовый hook для AWS
   - `s3_hook.S3Hook` — для Amazon S3
   - `gcp_api_base_hook.GoogleCloudBaseHook` — базовый hook для GCP

3. **API и веб-сервисы**:
   - `http_hook.HttpHook` — для HTTP-запросов
   - `slack_hook.SlackHook` — для Slack

**Безопасность Connections**:

1. **Шифрование паролей**:
   - Пароли и секреты в подключениях хранятся в зашифрованном виде
   - Для шифрования используется ключ, заданный в параметре `fernet_key` в airflow.cfg

2. **Внешние системы управления секретами**:
   - Интеграция с Vault, AWS Secrets Manager, GCP Secret Manager и другими
   - Настройка в файле airflow.cfg

## Что такое сенсоры?

**Сенсоры (Sensors)** в Airflow — это специальный тип операторов, которые периодически проверяют выполнение определенного условия и продолжают выполнение только когда условие будет выполнено. Сенсоры полезны для организации зависимостей от внешних событий, таких как появление файла, завершение задачи в другом DAG, или доступность внешнего сервиса.

**Основные характеристики сенсоров**:

1. **Принцип работы**:
   - Проверка условия через равные промежутки времени
   - Ожидание до выполнения условия или до истечения таймаута
   - Освобождение слота выполнения во время ожидания (в режиме reschedule)

2. **Режимы работы**:
   - `poke` (по умолчанию) — постоянно проверяет условие, занимая слот воркера
   - `reschedule` — освобождает слот воркера между проверками, экономя ресурсы

3. **Параметры сенсора**:
   - `mode` — режим работы (poke или reschedule)
   - `poke_interval` — интервал между проверками в секундах
   - `timeout` — максимальное время ожидания
   - `soft_fail` — определяет, должен ли сенсор отмечаться как failed или skipped при таймауте
   - `exponential_backoff` — увеличение интервала между проверками экспоненциально

**Основные типы сенсоров в Airflow**:

1. **FileSensor** — проверяет наличие файла в файловой системе:
   ```python
   from airflow.sensors.filesystem import FileSensor
   
   wait_for_file = FileSensor(
       task_id='wait_for_file',
       filepath='/path/to/file.csv',
       poke_interval=60,  # проверка каждую минуту
       timeout=60 * 60 * 5,  # максимальное время ожидания - 5 часов
       mode='reschedule',  # освобождать слот воркера между проверками
       dag=dag,
   )
   ```

2. **S3KeySensor** — проверяет наличие файла в Amazon S3:
   ```python
   from airflow.providers.amazon.aws.sensors.s3 import S3KeySensor
   
   wait_for_s3_file = S3KeySensor(
       task_id='wait_for_s3_file',
       bucket_key='data/{{ ds }}/my_file.csv',
       bucket_name='my-bucket',
       aws_conn_id='aws_default',
       poke_interval=300,  # проверка каждые 5 минут
       timeout=60 * 60 * 12,  # таймаут 12 часов
       dag=dag,
   )
   ```

3. **ExternalTaskSensor** — ожидает завершения задачи в другом DAG:
   ```python
   from airflow.sensors.external_task import ExternalTaskSensor
   
   wait_for_other_dag = ExternalTaskSensor(
       task_id='wait_for_other_dag',
       external_dag_id='other_dag',
       external_task_id='final_task',  # конкретная задача или None для всего DAG
       execution_delta=timedelta(hours=1),  # смещение времени выполнения (если DAG-и запускаются с разным расписанием)
       allowed_states=['success'],  # ожидать только успешного завершения
       failed_states=['failed', 'skipped'],  # состояния, при которых сенсор завершится неудачно
       poke_interval=60 * 10,  # проверка каждые 10 минут
       mode='reschedule',
       dag=dag,
   )
   ```

4. **HttpSensor** — проверяет HTTP-эндпоинт:
   ```python
   from airflow.providers.http.sensors.http import HttpSensor
   
   wait_for_api = HttpSensor(
       task_id='wait_for_api',
       http_conn_id='api_endpoint',
       endpoint='/health',
       request_params={},
       response_check=lambda response: response.status_code == 200,
       poke_interval=60,
       timeout=60 * 30,
       dag=dag,
   )
   ```

5. **SqlSensor** — ожидает результатов SQL-запроса:
   ```python
   from airflow.providers.common.sql.sensors.sql import SqlSensor
   
   wait_for_data = SqlSensor(
       task_id='wait_for_data',
       conn_id='postgres_conn',
       sql="SELECT COUNT(*) FROM table WHERE date = '{{ ds }}' AND status = 'ready'",
       success=lambda cnt: cnt[0][0] > 0,  # успех, если результат больше 0
       poke_interval=60 * 5,
       timeout=60 * 60 * 2,
       dag=dag,
   )
   ```

**Практические примеры использования сенсоров**:

1. **Ожидание загрузки данных перед обработкой**:
   ```python
   # Ожидание появления файла во внешней системе
   wait_for_data = S3KeySensor(
       task_id='wait_for_data',
       bucket_key='raw/{{ ds }}/data.csv',
       bucket_name='data-lake',
       aws_conn_id='aws_conn',
       poke_interval=300,
       timeout=60 * 60 * 12,
       mode='reschedule',
       dag=dag,
   )
   
   # Задача обработки, которая выполнится только после появления файла
   process_task = PythonOperator(
       task_id='process_data',
       python_callable=process_data_function,
       op_kwargs={'date': '{{ ds }}'},
       dag=dag,
   )
   
   wait_for_data >> process_task
   ```

2. **Координация между DAG-ами**:
   ```python
   # DAG 1: data_ingestion_dag.py
   ingest_task = PythonOperator(
       task_id='ingest_data',
       python_callable=ingest_function,
       dag=dag1,
   )
   
   # DAG 2: data_processing_dag.py
   wait_for_ingestion = ExternalTaskSensor(
       task_id='wait_for_ingestion',
       external_dag_id='data_ingestion_dag',
       external_task_id='ingest_data',
       execution_date_fn=lambda dt: dt,  # использовать ту же дату выполнения
       dag=dag2,
   )
   
   process_task = PythonOperator(
       task_id='process_data',
       python_callable=process_function,
       dag=dag2,
   )
   
   wait_for_ingestion >> process_task
   ```

3. **Мониторинг системы и обработка ошибок**:
   ```python
   # Сенсор для проверки доступности сервиса
   check_service = HttpSensor(
       task_id='check_service',
       http_conn_id='service_api',
       endpoint='/health',
       poke_interval=60,
       timeout=60 * 10,
       soft_fail=True,  # не прерывать DAG при таймауте
       dag=dag,
   )
   
   # Основная задача
   main_task = PythonOperator(
       task_id='main_task',
       python_callable=main_function,
       dag=dag,
   )
   
   # Альтернативная задача при недоступности сервиса
   fallback_task = PythonOperator(
       task_id='fallback_task',
       python_callable=fallback_function,
       trigger_rule='one_failed',  # выполнится, если check_service завершится неудачно
       dag=dag,
   )
   
   check_service >> main_task
   check_service >> fallback_task
   ```

**Рекомендации по использованию сенсоров**:

1. **Выбор режима работы**:
   - Используйте `mode='reschedule'` для длительных ожиданий
   - Режим `poke` подходит для коротких и срочных проверок

2. **Настройка интервалов и таймаутов**:
   - Выбирайте `poke_interval` в зависимости от срочности и нагрузки
   - Устанавливайте разумные `timeout` для предотвращения вечных блокировок
   - Используйте `exponential_backoff=True` для снижения числа проверок со временем

3. **Обработка таймаутов**:
   - Настройте `soft_fail=True`, если сенсор не критичен для выполнения DAG
   - Создавайте альтернативные пути выполнения с помощью trigger rules

4. **Оптимизация ресурсов**:
   - Создайте отдельный пул для сенсоров с ограниченным числом слотов
   - Используйте `mode='reschedule'` для экономии ресурсов
   - Группируйте проверки связанных условий в одном сенсоре

5. **Создание собственных сенсоров**:
   - Для специфических проверок создавайте кастомные сенсоры, наследуясь от BaseSensorOperator:
     ```python
     from airflow.sensors.base import BaseSensorOperator
     
     class CustomSensor(BaseSensorOperator):
         def __init__(self, custom_param, *args, **kwargs):
             super().__init__(*args, **kwargs)
             self.custom_param = custom_param
         
         def poke(self, context):
             # Логика проверки условия
             # Возвращает True, если условие выполнено, иначе False
             return check_condition(self.custom_param)
     ```
